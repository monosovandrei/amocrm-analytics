## Project Communication Rules

- Пользователь не технарь: объяснять простым русским языком.
- Отвечать максимально четко, кратко и точно.
- Не перегружать техническими деталями, командами и терминами без необходимости.
- Если техническая команда нужна, сначала объяснить простыми словами, зачем она нужна.
- Избегать AI-slop: не писать общие рассуждения, воду и лишние варианты.
- Перед ответом проверять, что в нем есть конкретный результат, понятное действие и нет лишнего.

## Frontend Anti-Slop Gate

Проект `amocrm-analytics` — рабочая CRM/аналитика, не промо-лендинг и не декоративный SaaS-макет. Любой новый UI-элемент обязан иметь одну из ролей: действие, навигация, фильтр, статус, ввод данных, таблица, график, рабочая метрика или обязательный контекст для решения.

Запрещено добавлять:

- декоративные `pill`, `chip`, `summary`, `badge`, карточки и мини-счётчики без действия или аналитической ценности;
- дублирующие счётчики в шапке, если те же данные уже есть в заголовке секции, таблице или виджете;
- новые визуальные паттерны ради заполнения пустого места;
- случайные градиенты, glow, blobs, orbs, bokeh, glassmorphism и похожий AI-декор;
- англоязычные видимые подписи в русскоязычном интерфейсе без причины;
- смешивание разных структур для однотипных сущностей: если Sales/CSM/Прогноз являются разделами, они должны быть оформлены одним компонентным паттерном.

Перед финальным ответом после frontend-изменений выполнить:

1. Проверить, что каждый новый блок можно объяснить через пользовательский сценарий.
2. Проверить, что новый блок не дублирует уже существующую информацию.
3. Запустить `npm run frontend:gate`.
4. Если менялся экран, открыть его и проверить desktop/mobile визуально: нет мусорных плашек, дублей, странных подписей и декоративной карточности.

Если `npm run frontend:gate` падает, финальный ответ запрещён до исправления или явно обоснованного `slop-check: allow <rule>`.
