# Стандарт подготовки проекта к ИТ/ИБ-согласованию

Документ описывает, как должен быть оформлен проект перед передачей в ИТ, автоматизацию или ИБ, чтобы первичная проверка не упиралась в базовые недочеты.

Цель: передавать не прототип, а проверяемый технический пакет, который можно развернуть, проанализировать и безопасно допустить к пилоту или production.

## 1. Что должно быть понятно с первого экрана репозитория

В корне проекта должен быть `README.md`, из которого сразу ясно:

- что делает проект;
- для кого он предназначен;
- какие внешние системы использует;
- какие данные читает, хранит и изменяет;
- есть ли запись во внешние системы или только чтение;
- как проект запускается локально;
- как проект разворачивается через Docker;
- какие переменные окружения нужны;
- какие проверки нужно выполнить перед запуском;
- какие ограничения есть для пилота и production.

Нельзя оставлять README в формате "npm install / npm start" без описания данных, рисков и контура размещения.

## 2. Обязательная структура репозитория

Минимальная структура:

```txt
.
├── README.md
├── .env.example
├── .gitignore
├── docker/
│   ├── docker-compose.yml
│   └── Dockerfile...
├── docs/
│   ├── IT_REVIEW.md
│   ├── DEPLOYMENT.md
│   ├── SECURITY.md
│   └── RECHECK_REPORT.md
├── src/ или apps/
├── migrations/ или prisma/migrations/
├── package.json / pyproject.toml / go.mod / другое
└── tests/
```

Если проект использует БД, миграции обязательны. Схема "приложение само как-нибудь создаст таблицы" для согласования не подходит.

## 3. Документы, которые должны быть приложены

### `README.md`

Основной документ для разработчиков и проверяющих.

Должен содержать:

- назначение проекта;
- архитектуру на 5-10 строк;
- список внешних интеграций;
- список команд запуска;
- список команд проверки;
- ссылку на документы в `docs/`;
- production-ограничения.

### `docs/IT_REVIEW.md`

Документ специально для ИТ/ИБ.

Должен содержать:

- назначение системы;
- контур размещения;
- какие сетевые порты используются;
- какие исходящие запросы выполняются;
- какие входящие endpoint-ы есть;
- какие данные хранятся в БД;
- какие секреты нужны;
- как устроены роли и доступы;
- есть ли журнал действий;
- есть ли внешняя аналитика, CDN, AI-сервисы, телеметрия;
- условия безопасного пилота;
- что система точно не делает.

### `docs/DEPLOYMENT.md`

Инструкция развертывания.

Должна содержать:

- требования к серверу;
- версии Docker/Node/Python/БД;
- порядок заполнения `.env`;
- команды сборки;
- команды запуска;
- команды применения миграций;
- команды проверки health endpoint;
- порядок обновления версии;
- порядок отката;
- где смотреть логи.

### `docs/SECURITY.md`

Описание защиты.

Должно содержать:

- модель ролей;
- аутентификацию;
- rate limiting;
- хранение паролей;
- хранение токенов;
- CORS;
- HTTP security headers;
- правила доступа к БД;
- правила публикации наружу;
- требования к TLS/VPN/reverse proxy;
- secret rotation;
- backup/restore;
- аудит действий.

### `docs/RECHECK_REPORT.md`

Короткий отчет перед повторной или финальной передачей.

Должен содержать:

- список замечаний ИТ;
- статус по каждому замечанию;
- что именно исправлено;
- какие проверки выполнены;
- результаты проверок;
- остаточные эксплуатационные условия, если они не относятся к коду.

## 4. Переменные окружения

В репозитории должен быть `.env.example`.

В нем должны быть только имена переменных и безопасные примеры:

```env
NODE_ENV=production
APP_PORT=4000
DATABASE_URL=postgresql://app_user:change_me@postgres:5432/app_db
JWT_SECRET=change_me_min_32_chars
ENCRYPTION_KEY=change_me_64_hex_chars
EXTERNAL_API_CLIENT_ID=change_me
EXTERNAL_API_CLIENT_SECRET=change_me
```

Запрещено:

- коммитить реальный `.env`;
- коммитить токены;
- коммитить пароли;
- коммитить дампы БД;
- коммитить реальные клиентские данные;
- писать секреты в README, issue, комментариях или docker-compose напрямую.

Перед передачей нужно выполнить secret scan.

Пример проверки:

```bash
rg -n "gho_|github_pat_|sk-[A-Za-z0-9]|xox[baprs]-|AKIA[0-9A-Z]{16}|BEGIN (RSA|OPENSSH|EC) PRIVATE KEY|PASSWORD=|SECRET=|TOKEN=|API_KEY=" -g '!node_modules' -g '!.git' -g '!dist' .
```

## 5. Docker и сетевой контур

Проект должен подниматься через Docker без ручного шаманства.

Обязательные требования:

- `docker compose build` проходит без ошибок;
- `docker compose up -d` поднимает все сервисы;
- health endpoint отвечает `200`;
- миграции применяются на пустой БД;
- приложение не требует локальных файлов, которых нет в репозитории;
- Postgres/MySQL/Redis не публикуются наружу без необходимости;
- внутренние сервисы доступны только внутри Docker network;
- Web/API публикуются только там, где это осознанно нужно.

Для внутреннего пилота безопаснее привязывать порты к localhost:

```yaml
ports:
  - "127.0.0.1:4000:4000"
```

Если доступ нужен пользователям, наружу должен смотреть reverse proxy с TLS, allowlist/VPN и логированием, а не база данных или внутренние сервисы.

## 6. База данных и миграции

Если проект использует БД, должны быть:

- миграции;
- понятная схема данных;
- команда применения миграций;
- команда seed, если нужен первичный админ;
- описание, какие таблицы содержат чувствительные данные;
- backup/restore инструкция;
- запрет публикации БД в интернет.

Проверка перед передачей:

```bash
docker compose down -v
docker compose up -d
```

После этого нужно убедиться, что приложение стартует на пустой БД и применяет миграции без ручных действий.

## 7. Аутентификация и доступы

Минимальные требования:

- пароли хранятся только в виде hash;
- JWT/session secret обязателен и не имеет дефолтного production-значения;
- есть роли пользователей;
- admin endpoints защищены отдельно;
- forbidden/unauthorized работают корректно;
- выключенные пользователи не могут войти;
- есть rate limit на login;
- есть глобальный базовый rate limit на API;
- CORS ограничен разрешенными origin-ами.

Нельзя:

- оставлять hardcoded admin password;
- делать общий пароль для всех;
- открывать admin API без роли;
- полагаться только на скрытие кнопок во frontend.

## 8. Журнал действий

Для корпоративного согласования нужен audit log.

Минимально логировать:

- успешный вход;
- неуспешный вход;
- создание/изменение/блокировку пользователя;
- изменение ролей;
- изменение настроек интеграций;
- запуск синхронизаций;
- импорт/экспорт данных;
- создание/изменение/удаление отчетов или критичных сущностей;
- ошибки интеграций.

В audit log нельзя писать:

- пароли;
- access token;
- refresh token;
- private key;
- значения secret-переменных;
- полные payload-ы с лишними персональными данными.

## 9. Внешние сервисы и телеметрия

В `docs/IT_REVIEW.md` нужно явно указать:

- какие внешние API вызываются;
- какие данные туда отправляются;
- используется ли AI/LLM;
- используется ли внешняя аналитика;
- используются ли CDN;
- используются ли внешние шрифты;
- есть ли Sentry/PostHog/GA/Amplitude/Meta Pixel/Яндекс Метрика;
- есть ли исходящие webhooks.

Если проект не использует AI или внешнюю аналитику, это нужно написать явно.

Пример формулировки:

```txt
Проект не использует LLM, AI-агентов, OpenAI, Anthropic, LangChain и аналогичные сервисы.
Данные не отправляются во внешний AI-контур.
Внешняя web-аналитика и телеметрия не подключены.
```

## 10. Работа с чувствительными данными

Для каждого типа данных нужно понимать:

- источник;
- зачем данные нужны;
- где они хранятся;
- кто имеет доступ;
- сколько они хранятся;
- можно ли их удалить;
- передаются ли они наружу.

Если сервис делает копию данных из CRM/ERP/1C, это нужно прямо описать как отдельный чувствительный контур.

## 11. Frontend

Frontend должен быть не демонстрационной картинкой, а рабочим интерфейсом.

Минимальные требования:

- нет критичных console errors;
- есть loading state;
- есть empty state;
- есть error state;
- длинные строки не ломают layout;
- таблицы выдерживают много строк;
- формы показывают ошибки рядом с полями;
- кнопки имеют disabled/loading state;
- интерфейс адаптирован под desktop и mobile/tablet, если это требуется;
- admin-действия доступны только пользователям с правами;
- frontend не содержит секретов.

Если проект внутренний, интерфейс должен быть рабочим и плотным, без лендинговой декоративности.

## 12. Backend/API

Минимальные требования:

- DTO validation включен;
- неизвестные поля запрещены или игнорируются осознанно;
- ошибки не раскрывают stack trace в production;
- health endpoint есть;
- CORS настроен;
- Helmet/security headers включены для HTTP API;
- rate limit включен;
- права проверяются на backend;
- внешние API вызываются с timeout/retry policy;
- секреты не логируются;
- входящие webhooks проверяются по подписи или secret URL.

## 13. Тесты

Минимальный набор тестов перед передачей:

- login success/fail;
- rate limit или хотя бы конфигурация throttling;
- проверка ролей;
- admin endpoint forbidden для обычного пользователя;
- миграции на пустой БД;
- критичная бизнес-логика;
- audit log для критичных действий;
- export/import, если есть;
- health endpoint.

Команды проверки должны быть описаны в README:

```bash
npm test
npm run typecheck
npm run build
npm audit --omit=dev
docker compose build
docker compose up -d
```

## 14. Production readiness checklist

Перед передачей ИТ все пункты должны быть закрыты:

- [ ] README описывает назначение, запуск, проверки и ограничения.
- [ ] Есть `.env.example` без реальных секретов.
- [ ] Реальный `.env` не отслеживается Git.
- [ ] Есть миграции БД.
- [ ] Docker build проходит.
- [ ] Docker up проходит на чистой БД.
- [ ] Health endpoint отвечает `200`.
- [ ] БД не опубликована наружу.
- [ ] Web/API опубликованы только в нужный контур.
- [ ] Login защищен rate limit.
- [ ] Admin endpoints защищены ролями.
- [ ] Пароли хранятся hash-ами.
- [ ] Токены/секреты шифруются или не хранятся.
- [ ] Audit log реализован.
- [ ] Audit log не содержит секретов.
- [ ] CORS ограничен.
- [ ] Security headers включены.
- [ ] `npm audit --omit=dev` или аналог чистый.
- [ ] Тесты проходят.
- [ ] Typecheck проходит.
- [ ] Production build проходит.
- [ ] Secret scan не находит реальных секретов.
- [ ] Документ `docs/IT_REVIEW.md` заполнен.
- [ ] Документ `docs/DEPLOYMENT.md` заполнен.
- [ ] Документ `docs/SECURITY.md` заполнен.
- [ ] Документ `docs/RECHECK_REPORT.md` заполнен перед повторной передачей.

## 15. Что нельзя отдавать на согласование

Нельзя передавать проект, если:

- он не стартует на чистой машине;
- нет миграций;
- база данных торчит наружу;
- в репозитории есть `.env`;
- есть реальные токены, пароли или клиентские данные;
- login можно перебирать без ограничений;
- admin API защищен только frontend-ом;
- нет описания внешних интеграций;
- нет инструкции развертывания;
- Docker не проверен;
- audit log заявлен, но не реализован;
- README не объясняет контур данных и риски;
- есть известные high/critical vulnerabilities.

## 16. Рекомендуемый порядок подготовки проекта

1. Привести README в нормальный вид.
2. Добавить `.env.example`.
3. Проверить `.gitignore`.
4. Удалить секреты и реальные данные из репозитория.
5. Добавить миграции.
6. Закрыть сетевые порты БД.
7. Проверить auth, роли и rate limit.
8. Добавить audit log.
9. Описать внешние интеграции.
10. Добавить Docker smoke на чистой БД.
11. Прогнать тесты, typecheck, build, audit.
12. Заполнить `docs/IT_REVIEW.md`.
13. Заполнить `docs/RECHECK_REPORT.md`.
14. Сделать финальный commit и tag/release candidate.

## 17. Формулировка для передачи ИТ

Можно использовать такой текст:

```txt
Передаем проект на техническую проверку.

В репозитории приложены:
- README с назначением, запуском и проверками;
- docs/IT_REVIEW.md для ИТ/ИБ;
- docs/DEPLOYMENT.md с порядком развертывания;
- docs/SECURITY.md с описанием защиты;
- docs/RECHECK_REPORT.md с результатами самопроверки.

Проверки перед передачей выполнены:
- tests passed;
- typecheck passed;
- production build passed;
- dependency audit clean;
- Docker build/up на чистой БД passed;
- secret scan не выявил реальных секретов.

Проект предполагается запускать во внутреннем контуре компании.
Публичная публикация Web/API допускается только через reverse proxy с TLS и правилами доступа.
База данных наружу не публикуется.
```

## 18. Главный принцип

ИТ не должно угадывать, безопасен проект или нет.

Проект должен сам отвечать на вопросы:

- что он делает;
- какие данные трогает;
- куда ходит по сети;
- что хранит;
- как защищен;
- как развернуть;
- как проверить;
- какие ограничения есть перед production.

Если эти ответы есть в коде, Docker и документации, шанс пройти согласование с первого раза резко выше.
