# Запрос в ИТ на выделение серверного контура для PulseBoard amoCRM Analytics

## 1. Что нужно развернуть

Просим выделить внутренний серверный контур для размещения сервиса аналитики продаж `PulseBoard amoCRM Analytics`.

Сервис предназначен для чтения данных из amoCRM, сохранения копии нужных данных в PostgreSQL и построения управленческих отчетов для РОПа.

Важно: сервис не должен размещаться на публичном сервере без VPN/reverse proxy и правил доступа. База данных не должна быть доступна из интернета.

## 2. Назначение сервера

На сервере будут размещены:

- backend API сервиса;
- frontend web-интерфейс;
- PostgreSQL база данных;
- Docker Compose окружение;
- конфигурационные `.env`-секреты;
- логи приложения;
- резервные копии PostgreSQL.

## 3. Требуемый контур размещения

Рекомендуемый вариант:

- внутренний сервер компании;
- доступ только из корпоративной сети или через VPN;
- внешний доступ из интернета не требуется;
- публикация web-интерфейса допускается только через reverse proxy с TLS;
- доступ к PostgreSQL только локально на сервере или внутри Docker network.

Запрещено:

- публиковать PostgreSQL наружу;
- открывать API напрямую в интернет без TLS/VPN/allowlist;
- хранить `.env` в Git;
- использовать реальные секреты из примеров.

## 4. Минимальные ресурсы сервера

Для пилота:

- CPU: 2 vCPU;
- RAM: 4 GB;
- Disk: 50 GB SSD;
- OS: Linux LTS, предпочтительно Ubuntu Server 22.04/24.04 LTS;
- Docker Engine: актуальная stable-версия;
- Docker Compose Plugin: актуальная stable-версия.

Для production после пилота:

- CPU: 4 vCPU;
- RAM: 8 GB;
- Disk: от 100 GB SSD;
- отдельный backup storage;
- мониторинг диска, CPU, RAM и состояния контейнеров.

## 5. ПО, которое нужно установить

На сервере нужны:

- Docker Engine;
- Docker Compose Plugin;
- Git;
- OpenSSL;
- curl;
- утилиты администрирования PostgreSQL, если backup/restore будет выполняться с хоста.

Node.js на хосте не обязателен, если запуск идет только через Docker.

## 6. Сетевые порты

Внутри сервера/контейнеров:

- Web: `3000`;
- API: `4000`;
- PostgreSQL: `5432`.

Рекомендуемая публикация на хосте:

- Web: `127.0.0.1:3000`;
- API: `127.0.0.1:4000`;
- PostgreSQL: не публиковать наружу, либо только `127.0.0.1:5433` для локального администрирования.

Если пользователям нужен доступ по ссылке:

- настроить reverse proxy, например Nginx/Traefik;
- выдать внутренний DNS, например `pulseboard.company.local`;
- включить TLS-сертификат;
- проксировать Web/API только через reverse proxy;
- ограничить доступ корпоративной сетью, VPN или allowlist.

## 7. DNS и URL

Нужно выделить внутренний URL для пользователей:

```txt
https://pulseboard.<company-domain>
```

Или внутренний адрес:

```txt
https://pulseboard.company.local
```

API может быть опубликован под тем же доменом через path:

```txt
https://pulseboard.<company-domain>/api/v1
```

Либо отдельным внутренним URL:

```txt
https://pulseboard-api.<company-domain>/api/v1
```

## 8. TLS и reverse proxy

Нужно настроить:

- TLS-сертификат для web-интерфейса;
- HTTP to HTTPS redirect;
- проксирование запросов frontend/backend;
- лимит размера request body;
- access logs;
- error logs;
- security headers на уровне proxy, если принято в компании;
- ограничение доступа по VPN/allowlist, если сервис не должен быть доступен всем сотрудникам.

## 9. Доступы

Нужно завести отдельные доступы:

- отдельный Linux-пользователь или service account для эксплуатации;
- доступ разработчикам только при необходимости;
- SSH только по ключам;
- запрет SSH по паролю;
- доступ к серверу только из корпоративной сети/VPN;
- доступ к PostgreSQL только администраторам БД/сервера;
- отдельный администратор приложения для первичной настройки.

## 10. Переменные окружения и секреты

На сервере нужно создать `.env` на основе `.env.example`.

Обязательные секреты:

```env
POSTGRES_PASSWORD=<сгенерировать сильный пароль>
DATABASE_URL=postgresql://amocrm:<POSTGRES_PASSWORD>@postgres:5432/amocrm_analytics
JWT_SECRET=<сгенерировать строку минимум 32 символа>
CREDENTIALS_ENCRYPTION_KEY=<сгенерировать 64 hex символа>
AMOCRM_CLIENT_ID=<выдать из amoCRM интеграции>
AMOCRM_CLIENT_SECRET=<выдать из amoCRM интеграции>
WEB_ORIGIN=https://pulseboard.<company-domain>
VITE_API_URL=https://pulseboard.<company-domain>/api/v1
```

Требования:

- `.env` хранить только на сервере;
- права на файл ограничить;
- не отправлять `.env` в мессенджерах;
- не коммитить `.env` в Git;
- предусмотреть регламент ротации секретов.

## 11. PostgreSQL

Нужно развернуть PostgreSQL в Docker Compose или как managed/internal PostgreSQL по стандарту компании.

Требования:

- отдельная база данных под проект;
- отдельный пользователь БД под проект;
- сильный пароль;
- доступ только из backend/API контейнера или localhost;
- запрет доступа из интернета;
- регулярный backup;
- проверенный restore;
- мониторинг заполнения диска;
- хранение backup в отдельном защищенном месте.

В базе будут храниться копии данных из amoCRM:

- сделки;
- контакты;
- компании;
- телефоны;
- email;
- менеджеры;
- группы;
- воронки и этапы;
- суммы сделок;
- история этапов;
- задачи;
- примечания;
- события;
- пользовательские настройки отчетов;
- журнал действий.

БД нужно считать чувствительным контуром клиентских данных.

## 12. Backup и restore

Нужно настроить:

- ежедневный backup PostgreSQL;
- хранение backup минимум 14-30 дней;
- хранение backup отдельно от основного диска сервера;
- ограничение доступа к backup;
- регулярную проверку восстановления;
- инструкцию restore.

Минимальная политика для пилота:

```txt
Daily backup: 1 раз в сутки
Retention: 14 дней
Restore test: перед запуском пилота и далее по регламенту ИТ
```

Для production:

```txt
Daily backup: 1 раз в сутки
Retention: 30 дней или по политике компании
Restore test: регулярно по регламенту
```

## 13. Логи и мониторинг

Нужно обеспечить:

- мониторинг доступности Web;
- мониторинг доступности API health endpoint;
- мониторинг состояния Docker containers;
- мониторинг CPU/RAM/disk;
- мониторинг свободного места PostgreSQL volume;
- сбор логов backend;
- сбор логов reverse proxy;
- алерт при падении контейнеров;
- алерт при заполнении диска.

Health endpoint:

```txt
GET /api/v1/health
```

Ожидаемый ответ:

```json
{"status":"ok","service":"amocrm-analytics-api"}
```

## 14. amoCRM интеграция

Для пилота сервису нужны только права чтения amoCRM.

Минимально нужны права на чтение:

- аккаунта;
- пользователей и групп;
- воронок и этапов;
- сделок;
- контактов;
- компаний;
- задач;
- примечаний;
- событий;
- custom fields.

На первом этапе не выдавать сервису права записи в amoCRM.

Сервис не должен подключаться к 1С и не должен влиять на существующую интеграцию amoCRM и 1С.

## 15. Требования к безопасности

Нужно обеспечить:

- запуск только во внутреннем контуре;
- TLS для пользовательского web-доступа;
- доступ через VPN/корпоративную сеть;
- запрет публикации БД наружу;
- хранение секретов только на сервере;
- backup с ограниченным доступом;
- SSH только по ключам;
- firewall на уровне сервера;
- ограничение доступа к админке приложения;
- журналирование действий пользователей в приложении;
- ротацию секретов по регламенту компании.

## 16. Что проект не требует

Для текущего пилота не требуется:

- публичный интернет-доступ;
- отдельный Kubernetes-кластер;
- подключение к 1С;
- права записи в amoCRM;
- внешний AI-сервис;
- внешняя web-аналитика;
- публикация PostgreSQL наружу;
- CDN;
- отдельный object storage, если backup решается средствами ИТ.

## 17. Что нужно получить от ИТ

Просим предоставить:

- сервер или VM с указанными ресурсами;
- внутренний DNS/URL;
- TLS-сертификат или способ выпуска сертификата;
- доступ по SSH ответственному инженеру;
- установленный Docker и Docker Compose;
- firewall-правила;
- место для backup;
- регламент backup/restore;
- способ мониторинга;
- подтверждение, что PostgreSQL не доступен из внешней сети;
- список сотрудников/ролей, которым разрешен доступ к web-интерфейсу.

## 18. Критерии готовности серверного контура

Контур считается готовым, если:

- сервер доступен по SSH только разрешенным пользователям;
- Docker и Docker Compose установлены;
- проект собирается через Docker;
- `docker compose up -d` запускает Web/API/PostgreSQL;
- миграции БД применяются на пустой базе;
- `GET /api/v1/health` возвращает `200`;
- web-интерфейс доступен по внутреннему HTTPS URL;
- PostgreSQL не доступен извне;
- backup PostgreSQL настроен;
- restore-процедура описана;
- мониторинг и алерты настроены;
- реальные секреты лежат только на сервере.

## 19. Короткая формулировка запроса

```txt
Просим выделить внутренний серверный контур для сервиса PulseBoard amoCRM Analytics.

Нужен Linux-сервер/VM под Docker Compose:
- пилот: 2 vCPU, 4 GB RAM, 50 GB SSD;
- production: 4 vCPU, 8 GB RAM, от 100 GB SSD.

На сервере будут работать Web, API и PostgreSQL.
Сервис читает данные из amoCRM и хранит их копию в PostgreSQL, поэтому БД считать чувствительным контуром клиентских данных.

Требуется:
- доступ только из корпоративной сети/VPN;
- HTTPS URL для пользователей;
- reverse proxy с TLS;
- PostgreSQL не публиковать наружу;
- ежедневный backup PostgreSQL;
- мониторинг Web/API/PostgreSQL/Docker/disk;
- хранение `.env` только на сервере;
- SSH только по ключам;
- для amoCRM на пилоте выдать только права чтения;
- к 1С сервис не подключать.

Критерий готовности: проект запускается через Docker Compose, миграции применяются на пустой БД, API health отвечает 200, Web доступен по внутренней HTTPS-ссылке, БД недоступна из внешней сети, backup и мониторинг настроены.
```
